# Med Helper

Офлайн-справочник клинических протоколов. Мобильное PWA (Vue 3 + Vite),
публикуется как статический сайт на GitHub Pages, работает без сети после
первого открытия.

## Что это и, главное, чем это НЕ является

Приложение **отображает** протокол в структурированном виде: список секций,
якорная навигация, цветовая маркировка типов секций, поиск по открытому
документу.

Приложение **не** является системой поддержки принятия решений:

- оно ничего не вычисляет, не суммирует и не интерпретирует;
- оно не выдаёт вердиктов «показано / не показано», «назначить / не назначать»;
- отметки чек-листа — визуальные закладки, по ним не считается ни итог, ни
  процент, ни балл;
- ячейки таблиц — строки, а не числа: почвы для арифметики над дозировками в
  формате данных нет по построению.

Клиническое решение и ответственность за него остаются за врачом. При первом
запуске приложение показывает дисклеймер и не открывает контент без явного
подтверждения; та же строка постоянно висит в футере.

Медицинского текста в коде нет вообще — весь контент живёт в JSON-файлах
(см. [Как добавить протокол](#как-добавить-протокол) и
[Правила работы с контентом](#правила-работы-с-контентом)).

## Требования

- **Node 22 LTS** — версия зафиксирована в [`.nvmrc`](./.nvmrc)
  (`nvm use` подхватит её сам)
- **npm** (идёт вместе с Node)

Установка зависимостей:

```bash
nvm use          # необязательно, но так версия Node гарантированно верная
npm ci           # при первом запуске в новом клоне — или npm install
```

Бэкенда нет, базы нет, ключей и переменных окружения для локальной работы не
требуется.

## Команды

| Команда | Что делает |
|---------|------------|
| `npm run dev` | Dev-сервер Vite с HMR. Дополнительно включает runtime-валидацию открытого протокола против JSON Schema: расхождение видно и в консоли, и плашкой на экране протокола. Service worker в dev намеренно не регистрируется (он ломал бы HMR) |
| `npm run build` | Прод-сборка в `dist/`: приложение + service worker (Workbox) + web-манифест. Базовый путь берётся из переменной `BASE_PATH` (по умолчанию `/`) |
| `npm run preview` | Локально отдаёт уже собранный `dist/`. **Единственный способ проверить офлайн и баннер обновления** — в dev-режиме service worker выключен |
| `npm run test` | Vitest (happy-dom): рендер всех типов блоков и фоллбэка, эфемерность чек-листа, поиск, дисклеймер, лента секций, баннер обновления |
| `npm run type-check` | `vue-tsc` по коду приложения + `tsc` по конфигам сборки, оба в strict. Реестр блоков типизирован так, что забытый тип блока роняет именно эту команду |
| `npm run validate:data` | Ajv: обе JSON Schema + кросс-проверки реестра и файлов протоколов. Выводит `файл + путь до поля + сообщение`, при проблемах завершается кодом 1. **Запускать после любой правки JSON** |
| `npm run gen:icons` | Перегенерирует иконки PWA в `public/icons/` (192/512/maskable + favicon). Нужна только при смене палитры в `src/assets/theme.css`; готовые иконки лежат в репозитории |

В CI перед сборкой прогоняются `validate:data` и `test` — протокол с битыми
данными физически не может уехать в прод.

## Как добавить протокол

Это основной рабочий сценарий владельца. **Правок кода он не требует ни
одной** — ни компонентов, ни роутера, ни списка на главном экране.

### Шаг 1. Создать файл протокола

`src/data/protocols/<id>.json`, где `<id>` — латиница в нижнем регистре,
цифры и дефисы (`^[a-z0-9][a-z0-9-]*$`). **Имя файла обязано совпадать с полем
`id` внутри него.**

```json
{
  "$schema": "../schema/protocol.schema.json",
  "id": "moy-protokol",
  "title": "Название протокола",
  "version": "1.0",
  "sourceDocument": "Название и реквизиты исходного документа",
  "sourceDate": "2026-03-15",
  "lastReviewed": "2026-03-20",
  "sections": [
    {
      "id": "razdel-1",
      "title": "Раздел 1",
      "kind": "default",
      "blocks": [
        { "type": "text", "body": "Текст абзаца." }
      ]
    }
  ]
}
```

Строка `"$schema"` не обязательна для приложения, но её стоит ставить: с ней
редактор (VS Code, IDEA) подсказывает поля и подсвечивает ошибки прямо при
наборе.

#### Поля прослеживаемости — обязательны

Без любого из четырёх полей протокол не проходит валидацию и не собирается.
Все четыре всегда видны на экране протокола, без раскрытия и прокрутки.

| Поле | Что значит |
|------|------------|
| `version` | Версия самого протокола (не приложения). Врач по ней понимает, ту ли редакцию он читает |
| `sourceDocument` | Из какого документа перенесён текст: название, номер, издавший орган — так, чтобы можно было найти оригинал |
| `sourceDate` | Дата исходного документа (ISO, `YYYY-MM-DD`) |
| `lastReviewed` | Дата, когда врач **вручную сверил** содержимое JSON с оригиналом (ISO). Ставится только после такой сверки — см. [правила работы с контентом](#правила-работы-с-контентом) |

#### Секции

| Поле | Обязательно | Что значит |
|------|-------------|------------|
| `id` | да | Якорь навигации, тот же формат, что у `id` протокола; уникален внутри протокола |
| `title` | да | Заголовок секции и подпись на чипе ленты навигации |
| `kind` | нет (по умолчанию `default`) | Цветовая маркировка: `default`, `inclusion` (показания), `exclusion-absolute` (абсолютные противопоказания), `exclusion-relative` (относительные противопоказания) |
| `blocks` | да | Содержимое секции |

### Шаг 2. Зарегистрировать протокол в реестре

В `src/data/protocols/index.json` добавить запись. Порядок записей в файле =
порядок протоколов на главном экране.

```json
{
  "$schema": "../schema/index.schema.json",
  "protocols": [
    {
      "id": "moy-protokol",
      "title": "Название протокола",
      "version": "1.0",
      "sourceDate": "2026-03-15"
    }
  ]
}
```

`title`, `version` и `sourceDate` продублированы намеренно: главный экран
показывает список, не загружая все файлы протоколов. **Значения обязаны
совпадать с файлом протокола буква в букву** — расхождение ловит валидатор:

```
src/data/protocols/moy-protokol.json /version — рассинхрон с реестром:
  index.json = "1.0", moy-protokol.json = "1.1"
```

### Шаг 3. Проверить

```bash
npm run validate:data
```

Ожидаемый вывод:

```
validate:data — OK: реестр и N протокол(ов) валидны,
биекция реестр↔файлы соблюдена, метаданные согласованы.
```

Валидатор проверяет: обе схемы; что у каждой записи реестра есть файл и у
каждого файла есть запись; равенство `title`/`version`/`sourceDate`; равенство
`id` имени файла; уникальность `section.id`.

Затем `npm run dev` — протокол уже в списке на главном экране. Ничего больше
делать не нужно.

## Типы блоков v1

Блок — атомарная единица отображения; тип задаётся полем `type`. Во всех
известных типах действует `additionalProperties: false`: опечатка в имени поля
будет ошибкой валидации, а не молча проигнорированным полем.

### `text` — абзацы

Абзацы разделяются пустой строкой (`\n\n`).

```json
{ "type": "text", "body": "Текст абзаца 1.\n\nТекст абзаца 2." }
```

### `warning` — акцентная плашка-предупреждение

```json
{ "type": "warning", "body": "Текст предупреждения." }
```

### `timer-note` — статичная заметка о временном окне

`label` необязателен. Это **заметка, а не таймер**: никакого отсчёта времени в
приложении нет и не будет.

```json
{ "type": "timer-note", "label": "Метка окна", "body": "Текст примечания." }
```

### `criteria-list` — список критериев с цветовой маркировкой

`kind` обязателен и принимает одно из трёх значений: `inclusion`,
`exclusion-absolute`, `exclusion-relative` — от него зависит цвет маркеров.
Минимум один пункт.

```json
{
  "type": "criteria-list",
  "kind": "inclusion",
  "items": ["Критерий 1", "Критерий 2"]
}
```

### `dosage-table` — таблица

`title` необязателен. **Все ячейки — строки**, минимум одна колонка. На узком
экране таблица скроллится по горизонтали внутри себя, страница целиком не
едет.

```json
{
  "type": "dosage-table",
  "title": "Заголовок таблицы (пример)",
  "columns": ["Колонка 1", "Колонка 2"],
  "rows": [
    ["Строка 1", "Пример значения"],
    ["Строка 2", "Пример значения"]
  ]
}
```

### `checklist` — интерактивный чек-лист

Отметки живут только пока открыт протокол: закрыл и открыл снова — чисто.
Ничего по отметкам не считается и не показывается.

```json
{ "type": "checklist", "items": ["Пункт 1", "Пункт 2"] }
```

## Неизвестные типы блоков

Блок с типом, которого текущая сборка не знает, — **валидные данные**, а не
ошибка. Схема требует от блока только `type: string`.

Что происходит на экране: вместо содержимого рисуется заметная плашка

> **Неподдерживаемый тип блока: «totally-new-block»**
> Содержимое не отображено. Обновите приложение до версии, которая знает этот
> тип.

Остальной протокол при этом рендерится полностью, приложение не падает; в
dev-режиме в консоль дополнительно уходит предупреждение со списком
незнакомых типов.

Почему это важно: приложение и контент обновляются независимо. Данные, где
появился новый тип блока, попадут на устройство раньше кода, который умеет его
рисовать. Молчаливо пропустить такой блок нельзя — врач должен видеть, что в
этом месте протокола что-то есть и оно не показано. Видимый фоллбэк — это
контракт, а не заглушка «на всякий случай».

Добавление нового типа блока (уже с правкой кода) = ветка в
`src/data/schema/protocol.schema.json` → вариант union в
`src/types/protocol.ts` → компонент в `src/components/blocks/` → строка в
`src/components/blocks/registry.ts`. Забудете последнюю — упадёт
`npm run type-check`, а не прод.

## Деплой

Публикация полностью автоматическая:
[`.github/workflows/deploy.yml`](./.github/workflows/deploy.yml).

1. Push в `main` запускает GitHub Actions (есть и ручной запуск —
   *workflow_dispatch*).
2. Workflow: `npm ci` → `npm run validate:data` → `npm run test` →
   `npm run build` → публикация `dist/` через официальные Pages-экшены.
   Job деплоя зависит от job'а сборки, поэтому упавшая валидация или упавший
   тест физически не дают опубликовать сборку.
3. Готово: приложение на GitHub Pages, deep-link вида
   `.../#/protocol/demo-protocol` открывается напрямую (роутер работает на
   хешах — SPA-фоллбэк на статическом хостинге не нужен).

### Механизм BASE_PATH

На GitHub Pages сайт живёт не в корне домена, а на под-пути репозитория.
Единственное место, где этот под-путь вообще написан, — переменная окружения
в шаге сборки workflow:

```yaml
- name: Build
  run: npm run build
  env:
    BASE_PATH: /medical-refference/
```

`vite.config.ts` читает её как `base: process.env.BASE_PATH ?? '/'`. Локально
переменной нет — база `/`, всё работает из корня. В коде адрес хостинга не
зашит нигде; `start_url` и `scope` в web-манифесте относительные (`./`),
поэтому одинаково верны и локально, и на под-пути Pages, и в будущем
WebView Capacitor. Переехали в другой репозиторий — правится одна строка
в workflow.

Проверить сборку под под-путь локально:

```bash
BASE_PATH=/medical-refference/ npm run build
```

### Разово: включить GitHub Pages

Шаг `actions/configure-pages` с `enablement: true` пытается включить Pages сам
при первом прогоне. Если у стандартного `GITHUB_TOKEN` не хватит прав, шаг об
этом сообщит — тогда Pages включается один раз руками:
**Settings → Pages → Source: GitHub Actions**. Дальше workflow работает как
есть.

## Офлайн и обновления

- После первого открытия с сетью service worker прекэширует **всё**, из чего
  состоит приложение: HTML, JS, CSS, иконки, манифест. Контент протоколов
  отдельными файлами не запрашивается — JSON запекается в JS-чанки сборки,
  то есть тоже лежит в прекэше. Сетевых запросов приложение не делает вовсе:
  ни к API, ни к CDN, ни к шрифтам.
- Обновления — только по явному действию. Новый service worker остаётся в
  ожидании, а сверху появляется баннер **«Доступно обновление»** с кнопкой.
  Без нажатия версия остаётся прежней. Молчаливая подмена версии протоколов
  запрещена конституцией: врач обязан знать, что содержимое изменилось.
- Проверять и офлайн, и баннер нужно на `npm run preview` — в dev-режиме
  service worker отключён.

## Правила работы с контентом

Это не пожелания, а действующие правила проекта — они зафиксированы в
[конституции](./.specify/memory/constitution.md), принципы I–III и V.

1. **Медицинский контент живёт ТОЛЬКО в JSON** (`src/data/protocols/`).
   В компонентах Vue медицинского текста нет: они рендерят любой валидный
   протокол по общей схеме. Врач ревьюит контент как данные, не читая код.
2. **Контент переносится из документов владельца, а не генерируется.**
   AI-агент никогда не пишет клинические критерии, дозировки, временные окна и
   противопоказания из собственных знаний. Единственный допустимый источник —
   документ, предоставленный владельцем; формулировки переносятся с
   сохранением исходного текста. Неоднозначные, нечитаемые или противоречивые
   места помечаются TODO для ручной проверки — «догадываться» и дополнять
   пробелы запрещено.
3. **`lastReviewed` ставится только после сверки врачом.** Перенесённый
   контент с TODO-пометками не считается готовым. Дата в этом поле означает
   ровно одно: живой человек с медицинским образованием построчно сверил JSON
   с оригиналом именно в этот день.
4. **Никакой интерпретации.** В коде не появляются пороги, суммы, категории
   риска и вердикты. Интерпретационная таблица из документа может быть в
   протоколе — но как обычный статический блок контента, наравне с остальным
   текстом.
5. **Ноль сети и ноль телеметрии.** Никаких аккаунтов, аналитики,
   синхронизации и пуш-уведомлений. Единственное, что приложение хранит на
   устройстве, — флаг «дисклеймер подтверждён» в `localStorage`. За пределы
   устройства не уходит ничего.

Демо-протокол `src/data/protocols/demo-protocol.json` — намеренно пустышка
(«Критерий 1», «Пример дозировки»): он существует как образец структуры и
фикстура рендерера, а не как медицинский источник.
