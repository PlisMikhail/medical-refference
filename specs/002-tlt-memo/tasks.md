## Зависимости

```text
Phase 1 (T001–T002)
   └─> Phase 2 (T003–T006)          ← блокирует всё остальное
          ├─> Phase 3 US1 (T007 → T008 → T009 → T010 → T011)   ← MVP
          │      └─> Phase 4 US2 (T012 → T013 → T014 → T015 → T016)
          │             └─> Phase 5 US3 (T017 → T018 → T019 → T020)
          │                    └─> Phase 6 (T021 ‖ T022 → T023 → T024)
```

- **US1 самостоятельна** после Phase 2: показания и противопоказания — это уже
  работающая памятка.
- **US2 зависит от T007** (файл протокола), но не от содержания US1: технически
  её можно делать сразу после T008, если понадобится.
- **US3 идёт последней** только потому, что заполняет карту сверки по факту
  перенесённого. Пометка черновика (T017) может быть сделана в любой момент —
  и чем раньше, тем безопаснее.

## Параллельность

Фича почти линейна: одиннадцать задач из двадцати четырёх правят один и тот же
JSON-файл, и распараллеливать их означало бы получать конфликты на каждом шаге.
Реально параллельны только T021 и T022 — разные файлы, ни одной общей строки.

Это нормально для контентной фичи: узкое место здесь — внимательность при
переносе, а не пропускная способность.

## Стратегия

**MVP** — Phase 1 + Phase 2 + Phase 3 (T001–T011). На этом объёме памятка
отвечает на главный вопрос «можно ли этому пациенту ТЛТ», и её уже имеет смысл
показать врачу.

Дальше — Phase 4 (дозы и обвязка), Phase 5 (статус и карта сверки), Phase 6
(тесты, README, приёмка).

Фича считается завершённой в состоянии **черновика**: снятие пометки —
отдельное действие после врачебной сверки, три правки в данных, описанные
в [source-map.md](./source-map.md).
