# Quickstart: приёмка ТЛТ-памятки

Сценарии, доказывающие соответствие спеке. Ссылки: [spec.md](./spec.md),
[plan.md](./plan.md), [research.md](./research.md),
[source-map.md](./source-map.md).

## Предпосылки

- Всё из [quickstart фичи 001](../001-protocol-viewer/quickstart.md) (Node 22, `npm ci`).
- Для сценария 2 дополнительно нужен `source-documents/protocol.txt` — извлечённый
  текст исходного PDF. Файл вне гита; как его получить, описано в README.
  Без него сценарий 2 не запускается, остальные работают.

## Команды

| Команда | Что делает |
|---------|------------|
| `npm run validate:data` | схема + кросс-проверка реестра |
| `npm run verify:transcription` | дословность памятки против исходного текста |
| `npm run test` | Vitest, включая проверки состава памятки |
| `npm run type-check`, `npm run build` | сборка не сломана |
| `npm run preview` | прод-сборка, нужна для проверки офлайна |

## Сценарий 1. Памятка появилась и открывается (US1, FR-001/002; SC-001)

1. `npm run dev`, мобильный viewport ~380px.
2. Ожидаемо: в списке **одна** запись — памятка ТЛТ. Демо-протокол из приложения
   убран. Видны название, версия `1.0.0-draft` и дата источника.
3. Тап по памятке. Ожидаемо: протокол открылся; лента чипов содержит шесть
   клинических секций в порядке показания → противопоказания → дозировки →
   АД → нейровизуализация → после ТЛТ.
4. Считаем тапы от запуска приложения до списка противопоказаний. Ожидаемо:
   не больше двух (протокол → чип).

---

## Сценарий 2. Дословность доказана машинно (FR-009/010/013/014; SC-003/004)

1. `npm run verify:transcription`.
2. Ожидаемо: зелёный отчёт — каждая клиническая строка найдена в исходном тексте
   как непрерывный фрагмент; число проверенных строк и число исключений выведены.
3. **Контрольная проверка самой проверки** (иначе зелёный отчёт ничего не стоит):
   временно изменить в JSON одну цифру — например, `0,9 мг/кг` на `0,8 мг/кг`.
   Ожидаемо: скрипт краснеет и называет секцию, блок и строку. Откатить правку.
4. Второй контроль: временно поменять местами два слова внутри любого пункта.
   Ожидаемо: скрипт краснеет — непрерывность фрагмента нарушена. Откатить.

## Сценарий 3. Полнота переноса (FR-003/004; SC-002)

1. `npm run test`. Ожидаемо: тесты состава памятки зелёные — число позиций
   в показаниях и противопоказаниях совпадает с документом.
2. Открыть секцию показаний. Ожидаемо: присутствуют все подпункты 28.1–28.6,
   ни один не объединён с соседним.
3. Открыть секцию противопоказаний. Ожидаемо: одна секция, красная маркировка,
   все позиции п. 29. Отдельной секции «относительные противопоказания» нет —
   так и задумано (документ такого деления не содержит).

## Сценарий 4. Дозировки статичны (US2, FR-005; принцип I конституции)

1. Открыть секцию дозировок.
2. Ожидаемо: таблица с алтеплазой и тенектеплазой — доза на кг, предельная доза,
   форма выпуска.
3. Ожидаемо: **нет** поля ввода веса, нет рассчитанной дозы, нет никакой
   арифметики. Приложение показывает текст протокола и ничего не вычисляет.

## Сценарий 5. Черновой статус виден (US3, FR-016; SC-006)

1. Главный экран. Ожидаемо: версия памятки помечена как черновик (`1.0.0-draft`).
2. Открыть памятку. Ожидаемо: **первым** блоком — предупреждение о том, что
   транскрипция не сверена врачом и дата проверки является датой транскрипции.
3. Ожидаемо: врач видит это до того, как прочтёт первый клинический пункт.

## Сценарий 6. Прослеживаемость (FR-011/015; принцип V)

1. В открытой памятке проверить метаданные: версия, исходный документ, дата
   источника, дата проверки — все четыре видны.
2. Открыть любую секцию. Ожидаемо: в ней есть ссылка на номер пункта исходного
   документа.
3. Сломать синхронность: изменить `version` в `index.json`, не трогая файл
   протокола. `npm run validate:data` — ожидаемо: ошибка кросс-проверки.
   Починить.

## Сценарий 7. Поиск по памятке (FR-020)

1. Открыть памятку, ввести в поиск `тенектеплаз` в другом регистре.
2. Ожидаемо: совпадения подсвечены, переход между ними работает.
3. Ввести слово, встречающееся в нескольких секциях (например, `NIHSS`).
   Ожидаемо: находится во всех.

## Сценарий 10. Врачебная сверка (US3; SC-005)

1. Распечатать или открыть [source-map.md](./source-map.md).
2. Врач проходит по карте с документом в руках, отмечая расхождения.
3. Ожидаемо: сверка возможна без чтения кода приложения; каждая секция
   привязана к номеру пункта; список не-исходных строк конечен и прочитан
   целиком.
4. После сверки — три правки из раздела «После сверки» карты, затем
   `npm run validate:data` и `npm run verify:transcription` повторно.

### Состояние проверки

**Проверено без браузера, приёмка владельцем не проводилась.**

Что доказано машинно и перепроверено независимо от отчётов агентов:

1. **Инструмент дословности умеет краснеть** — одиннадцать контрольных случаев:
   изменённая цифра, переставленные слова, потерянное «не», склейка двух разных
   мест документа, попытка покрыть длинную строку коротким исключением, мёртвая
   запись в списке, пустая строка. Ловится всё.
2. **Аудит чисел** — все числовые значения памятки сверены с источником
   по множеству; каждое расхождение объяснено колонтитулом страницы, ни одного
   необъяснённого.
3. **Происхождение фрагментов** — 34 из 34 клинических строк подтверждены как
   взятые из своего пункта документа, а не из похожего места (32 однозначно,
   2 однословные ячейки таблицы присутствуют в нужном диапазоне).
4. **Привязка «препарат ↔ доза ↔ форма выпуска»** сверена вручную: это
   единственное, что проверка дословности пропустила бы по построению.
5. **Переход в чистовой статус** — три правки из раздела «После сверки»
   применены на копии данных: `validate:data`, `verify:transcription` и все 67
   тестов остаются зелёными, правок кода и тестов не требуется.

Требует живой проверки владельцем (сценарии 1, 3, 5, 7, 8): вид памятки на
экране, читаемость длинного подпункта 28.1, работа ленты чипов на шести
секциях, поиск, офлайн на прод-сборке.

Требует врачебной сверки: см. [source-map.md](./source-map.md) — 39 числовых
значений, таблица дозировок и 10 открытых TODO.
