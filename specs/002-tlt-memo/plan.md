# Implementation Plan: Памятка по ВВ ТЛТ при остром ишемическом инсульте

**Branch**: `002-tlt-memo` | **Date**: 2026-07-21 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `/specs/002-tlt-memo/spec.md`

## Summary

Первый реальный клинический контент в приложении: раздел о внутривенной
тромболитической терапии из клинического протокола владельца переносится
в `src/data/protocols/tlt-ischemic-stroke.json` и регистрируется в реестре.
Кода приложения фича не касается.

Технически интересна здесь ровно одна вещь — **как доказать, что транскрипция
дословна**. Читать глазами и говорить «я сверил» недостаточно: цена ошибки
в дозировке или пороге — вред пациенту, а глаз на сороковом пункте устаёт.
Поэтому план строится вокруг машинной проверки: каждая клиническая строка JSON
обязана быть непрерывным фрагментом исходного текста при сравнении,
нечувствительном к дефектам извлечения PDF. Всё, что этой проверке не поддаётся,
попадает в явный список исключений с обоснованием — и это тот самый короткий
список, который врач читает при сверке вместо всего файла.

## Technical Context

**Language/Version**: изменений нет — TypeScript 5.9 / Vue 3.5 / Node 22 из фичи 001.
Артефакт фичи — данные (JSON) и один вспомогательный скрипт на Node (ESM).

**Primary Dependencies**: ничего нового. Проверочный скрипт использует только
встроенные модули Node (`node:fs`, `node:path`). Валидация схемы — существующий
`scripts/validate-protocols.mjs` на Ajv 8 (dev-зависимость).

**Storage**: файлы. `src/data/protocols/tlt-ischemic-stroke.json` (в гите),
`src/data/protocols/index.json` (правка), `source-documents/protocol.pdf`
и `source-documents/protocol.txt` (вне гита).

**Testing**: существующий Vitest; плюс два уровня проверки данных —
`npm run validate:data` (схема + кросс-проверка реестра, уже есть)
и `npm run verify:transcription` (новый, дословность против исходного текста).

**Target Platform**: не меняется — мобильный браузер, PWA, офлайн.

**Project Type**: контентное дополнение к принятому SPA.

**Performance Goals**: памятка не должна заметно утяжелить прекэш. Ориентир —
JSON до ~40 КБ; это порядок величины одного изображения и на офлайн-установку
не влияет.

**Constraints**: дословность (принцип II конституции) важнее краткости, читаемости
и любых соображений вёрстки. При конфликте выигрывает исходный текст.

**Scale/Scope**: один протокол, шесть клинических секций, ~30 переносимых пунктов
исходного документа.

## Constitution Check

*GATE: пройден до Phase 0, перепроверен после Phase 1.*

| Принцип | Как соблюдается | Проверяется чем |
|---------|-----------------|-----------------|
| I. Справочник, а не советчик | В памятке нет ни калькулятора, ни интерпретации. Дозировки — статическая таблица строк; схема запрещает в её ячейках что-либо, кроме строк, поэтому арифметика невозможна конструктивно | Ревью + `validate:data` |
| II. Запрет на генерацию контента (NON-NEGOTIABLE) | Каждая клиническая строка — непрерывный фрагмент исходного текста. Не-исходные строки перечислены поимённо в `transcription-exemptions.json` с обоснованием каждой | `verify:transcription` — машинно, а не «на глаз» |
| III. Данные отдельно от кода | Диффов в `src/` вне `src/data/protocols/` — ноль | `git diff --stat` в DoD |
| IV. Полный офлайн | Новый JSON попадает в прекэш тем же правилом, что и демо-протокол | Сценарий офлайна в quickstart |
| V. Прослеживаемость | `version`, `sourceDocument`, `sourceDate`, `lastReviewed` заполнены; кросс-проверка реестра ловит рассинхрон | `validate:data` |
| VI. Приватность | Фича не вводит сетевого взаимодействия | — |
| VII. Работоспособность web-сборки | `build`, `test`, `type-check` в DoD | CI |

**Нарушений нет.** Одно решение требует явной фиксации, см. Complexity Tracking:
в репозиторий добавляется скрипт `scripts/verify-transcription.mjs`. Кодом
приложения он не является, но формально это не «чистые данные».

**Structure Decision**: раскладка фичи 001 сохраняется без изменений. Новый
протокол — обычный файл рядом с демо-протоколом: именно это и проверяет
утверждение фичи 001 о том, что добавление протокола не требует кода.

Проверочный скрипт положен в `scripts/`, а не в `tests/`, сознательно: его вход —
`source-documents/protocol.txt`, файл вне репозитория. В `tests/` он падал бы
в CI и на чужой машине. Как ручной инструмент сверки, доступный владельцу
и врачу в любой момент, он на своём месте.

## Complexity Tracking

| Решение | Зачем | Почему не проще |
|---------|-------|-----------------|
| Скрипт `verify-transcription.mjs` в репозитории | Даёт машинное доказательство дословности и позволяет владельцу и врачу перепроверить транскрипцию в любой момент, в том числе после будущих правок | Альтернатива — «агент прочитал внимательно». Для дозировок и порогов это не доказательство: ошибка в одной цифре не отличима от опечатки и не ловится ревью. ~90 строк на встроенных модулях Node — цена, несопоставимая с ценой ошибки |
| Файл исключений `transcription-exemptions.json` | Делает конечным и обозримым список строк, которых в источнике нет (заголовки секций, ссылки на пункты, предупреждение о черновике) | Без него проверка либо ложно краснеет на каждом заголовке, либо её приходится ослаблять до бесполезности. Список исключений — это ровно то, что врач обязан прочитать глазами; всё остальное доказано машиной |

Обе позиции — инструментарий сверки, а не функциональность приложения. Ни одна
не попадает в сборку и не влияет на офлайн, размер бандла или путь к Capacitor.
