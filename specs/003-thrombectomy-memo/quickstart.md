# Quickstart: приёмка памятки по ВСТЭ

Сценарии, доказывающие соответствие спеке. Ссылки: [spec.md](./spec.md),
[plan.md](./plan.md), [research.md](./research.md),
[source-map.md](./source-map.md).

## Предпосылки

- Всё из [quickstart фичи 001](../001-protocol-viewer/quickstart.md) (Node 22, `npm ci`).
- Для сценария 3 дополнительно нужен `source-documents/protocol.txt` — тот же
  файл, что и в фиче 002, вне гита. Без него сценарий 3 не запускается,
  остальные работают.

## Команды

| Команда | Что делает |
|---------|------------|
| `npm run validate:data` | схема + кросс-проверка реестра, оба протокола |
| `npm run verify:transcription` | дословность обеих памяток против исходного текста |
| `npm run test` | Vitest, включая проверки состава обеих памяток |
| `npm run type-check`, `npm run build` | сборка не сломана |
| `npm run preview` | прод-сборка, нужна для проверки офлайна |

## Сценарий 1. Две памятки на главном экране (US3, FR-001; SC-010)

1. `npm run dev`, мобильный viewport ~380px.
2. Ожидаемо: в списке **две** записи — памятка ТЛТ и памятка ВСТЭ. У каждой
   видны название, версия и дата источника.
3. Ожидаемо: у ТЛТ версия `1.0.0`, у ВСТЭ — с суффиксом `-draft`.
4. Тап по памятке ТЛТ. Ожидаемо: открывается ровно то, что было принято врачом,
   без изменений. Назад — список на месте.

---

## Сценарий 2. Отбор по временному окну в два тапа (US1, FR-002/005/025; SC-001)

1. Открыть памятку ВСТЭ.
2. Ожидаемо: первая клиническая секция — показания (п. 31).
3. Считаем тапы от запуска приложения до критериев отбора в окне 6–24 часа.
   Ожидаемо: не больше двух (протокол → чип).
4. Ожидаемо: в этой секции — только позиции п. 33; позиций из пп. 32, 34, 35
   в ней нет.
5. Ожидаемо: в секции виден дословный текст отсылки к п. 35, и сам п. 35
   доступен отдельным чипом в той же ленте.

---

## Сценарий 3. Дословность доказана машинно (FR-013/014/016; SC-003/004)

1. `npm run verify:transcription`.
2. Ожидаемо: зелёный отчёт по **двум** протоколам; для каждого выведено число
   проверенных строк, найденных в источнике и покрытых исключениями.
3. **Контрольная проверка самой проверки:** временно изменить в JSON памятки
   ВСТЭ одну цифру — например, `ASPECT > 6` на `ASPECT > 5`. Ожидаемо: скрипт
   краснеет и называет секцию, блок и строку. Откатить правку.
4. Второй контроль: временно добавить в список исключений запись, которой нет
   в памятке. Ожидаемо: скрипт краснеет на неиспользованной записи. Откатить.

---

## Сценарий 4. Позиции не перепутаны между окнами (FR-005; SC-005)

Главный риск фичи: позиции пп. 32, 33 и 35 частично совпадают дословно, поэтому
позиция из чужого пункта пройдёт машинную проверку молча (см. R8 в research.md).

1. `npm run test`. Ожидаемо: тесты состава зелёные — в каждой из четырёх секций
   отбора ровно столько позиций, сколько в её пункте документа.
2. Открыть по очереди четыре секции отбора. Для каждой сверить с документом
   не только текст позиций, но и **их количество**.
3. Ожидаемо: позиция про ПМА есть только в секции 0–6 часов; отсылка к п. 35 —
   только в секции 6–24 часа; возрастные границы (< 80 / > 80 лет) — только
   в секции КТП 6–24 часа; коэффициент несоответствия > 1,8 — только в секции
   КТП 6–16 часов.

## Сценарий 5. Полнота переноса (FR-003/004/006–012; SC-002)

1. `npm run test`. Ожидаемо: счётчики позиций по всем секциям сходятся
   с документом.
2. Пройти по [source-map.md](./source-map.md) сверху вниз: каждая секция
   приложения имеет пункт документа, каждый пункт скоупа имеет секцию.
3. Ожидаемо: тринадцать секций в порядке из plan.md; ни одна не пуста.

---

## Сценарий 6. Лента чипов на тринадцати секциях (US2, FR-025/026)

1. Открыть памятку ВСТЭ на ширине ~380 px.
2. Ожидаемо: лента чипов прокручивается горизонтально, названия читаемы, ни одно
   не обрезано до неузнаваемости.
3. Ожидаемо: четыре секции отбора различаются по названию — окно и источник
   данных видны, не открывая секцию.
4. Тап по чипу в конце ленты. Ожидаемо: переход состоялся с первого раза,
   активный чип виден.
5. **Если что-то из этого разваливается** — это НЕ правится в этой фиче
   (FR-027). Заводится отдельная фича на ленту чипов; контент от неё
   не зависит.

## Сценарий 7. Черновой статус виден (US4, FR-023; SC-007)

1. Главный экран. Ожидаемо: версия памятки ВСТЭ помечена как черновик.
2. Открыть памятку. Ожидаемо: **первым** блоком — предупреждение о том, что
   транскрипция не сверена врачом.
3. Ожидаемо: врач видит это до того, как прочтёт первый клинический пункт.

---

## Сценарий 8. Прослеживаемость (FR-018/022; принцип V)

1. В открытой памятке проверить метаданные: версия, исходный документ, дата
   источника, дата проверки — все четыре видны.
2. Открыть любую секцию. Ожидаемо: в ней есть ссылка на номер пункта исходного
   документа.
3. Сломать синхронность: изменить `version` в `index.json`, не трогая файл
   протокола. `npm run validate:data` — ожидаемо: ошибка кросс-проверки.
   Починить.

---

## Сценарий 9. Поиск по памятке (FR-029)

1. Открыть памятку ВСТЭ, ввести в поиск `стент-ретривер` в другом регистре.
2. Ожидаемо: совпадения подсвечены, переход между ними работает.
3. Ввести `NIHSS`. Ожидаемо: находится во всех секциях, где встречается,
   включая все четыре секции отбора.

---

## Сценарий 10. Офлайн с двумя протоколами (FR-028; SC-008)

1. `npm run build && npm run preview`, открыть, дождаться установки SW.
2. Devtools → Network → Offline → полная перезагрузка.
3. Ожидаемо: **обе** памятки открываются и читаются целиком, поиск работает,
   внешних сетевых запросов ноль.

## Сценарий 11. Кода не прибавилось (FR-027; SC-009)

1. `git diff --stat` (или `git show --stat` после коммита).
2. Ожидаемо: в `src/` изменён только `src/data/protocols/index.json` и добавлен
   `src/data/protocols/vste-ischemic-stroke.json`. Компоненты, типы блоков
   и схема не тронуты.
3. Ожидаемо: `src/data/protocols/tlt-ischemic-stroke.json` не изменён ни на байт.
4. Вне `src/` допустимы: одна запись в массиве `TARGETS`
   `scripts/verify-transcription.mjs`, `tests/vste-memo.spec.ts`,
   `specs/003-thrombectomy-memo/`, `README.md`.

## Сценарий 12. Врачебная сверка (US4; SC-006)

1. Распечатать или открыть [source-map.md](./source-map.md).
2. Врач проходит по карте с документом в руках, отмечая расхождения.
3. Ожидаемо: сверка возможна без чтения кода; каждая секция привязана к номеру
   пункта; список не-исходных строк конечен и прочитан целиком.
4. Ожидаемо: врач отдельно проходит таблицу «четыре набора критериев поперёк» —
   это то, чего машинная проверка не доказывает.
5. После сверки — правки из раздела «После сверки» карты, затем
   `npm run validate:data` и `npm run verify:transcription` повторно.

## Definition of Done

- [ ] `npm run validate:data` зелёный на двух протоколах
- [ ] `npm run verify:transcription` зелёный на двух протоколах, и его
      контрольная проверка показала, что он умеет краснеть
- [ ] `npm run test`, `npm run type-check`, `npm run build` зелёные
- [ ] В `src/` изменены только данные; файл памятки ТЛТ не тронут
- [ ] `source-map.md` заполнена и готова к печати
- [ ] Черновой статус виден на главном экране и первым блоком памятки
