# Feature Specification: Просмотрщик клинических протоколов (offline)

**Feature Branch**: `001-protocol-viewer`

**Created**: 2026-07-21

**Status**: Draft

**Input**: User description: "Мобильный офлайн-справочник клинических протоколов. Пользователь — врач-невролог, сценарий использования — цейтнот (приёмное отделение): минимум тапов до нужного раздела, крупные тач-таргеты, тёмная тема по умолчанию. Главный экран — список протоколов из реестра; экран протокола — секции с якорной навигацией и цветовой маркировкой; генерический рендерер JSON→UI с типами блоков v1; поиск по открытому протоколу; дисклеймер при первом запуске; демо-протокол с placeholder-контентом; runtime-валидация JSON в dev-режиме. Вне скоупа: реальный контент ТЛТ, iOS, любая сеть."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Открыть протокол и прочитать его содержимое (Priority: P1)

Врач открывает приложение, видит список доступных протоколов (название, версия,
дата источника) и открывает нужный. Экран протокола показывает все секции
документа: текстовые блоки, предупреждения, временные окна, списки критериев,
таблицы дозировок и интерактивные чек-листы — всё отрендерено из данных
протокола, без «зашитого» в приложение медицинского текста. Метаданные
прослеживаемости протокола (версия, исходный документ, дата источника, дата
последней проверки) видны на экране протокола.

**Why this priority**: это ядро продукта — без просмотра содержимого протокола
остальные возможности бессмысленны. Уже одна эта история даёт работающий MVP
на демо-протоколе.

**Independent Test**: с демо-протоколом в реестре открыть приложение, перейти
в протокол и убедиться, что каждый тип блока v1 отображается, а метаданные
прослеживаемости видны.

**Acceptance Scenarios**:

1. **Given** реестр содержит демо-протокол, **When** врач открывает приложение,
   **Then** на главном экране виден список протоколов, у каждого — название,
   версия и дата источника.
2. **Given** главный экран открыт, **When** врач тапает по протоколу,
   **Then** открывается экран протокола со всеми его секциями и метаданными
   прослеживаемости (version, sourceDocument, sourceDate, lastReviewed).
3. **Given** протокол содержит блоки всех типов v1 (text, warning, timer-note,
   criteria-list, dosage-table, checklist), **When** экран протокола отображён,
   **Then** каждый блок отрендерен согласно своему типу, ни один не пропущен.
4. **Given** протокол содержит блок неизвестного типа, **When** экран протокола
   отображён, **Then** на месте блока виден явный фоллбэк «неподдерживаемый тип
   блока» с указанием типа; блок не скрыт молча.
5. **Given** открыт блок-чеклист, **When** врач отмечает пункты,
   **Then** отметки видны; **When** протокол закрыт и открыт снова,
   **Then** отметки сброшены (состояние не сохраняется).

### User Story 2 - Мгновенно попасть в нужную секцию и различить её тип (Priority: P2)

В цейтноте врач открывает протокол и должен за секунды оказаться в нужной
секции: доступна якорная навигация по секциям (оглавление или закреплённые
табы), а секции с показаниями, абсолютными и относительными противопоказаниями
маркированы цветом и различаются мгновенно, без вчитывания в заголовки.
Интерфейс — в тёмной теме по умолчанию, с крупными тач-таргетами.

**Why this priority**: скорость доступа к секции — главное требование сценария
«приёмное отделение»; но без работающего просмотра (P1) навигация бессмысленна.

**Independent Test**: на демо-протоколе с ≥5 секциями, включая все три вида
критериев, перейти в каждую секцию через навигацию и визуально различить типы
секций не читая заголовков.

**Acceptance Scenarios**:

1. **Given** открыт протокол с несколькими секциями, **When** врач использует
   навигацию по секциям, **Then** переход к любой секции выполняется в один тап
   из видимой навигации.
2. **Given** протокол содержит секции «показания», «абсолютные
   противопоказания», «относительные противопоказания», **When** экран
   отображён, **Then** эти три типа секций различимы мгновенно по цветовой
   маркировке единой темы приложения.
3. **Given** приложение открыто впервые, **When** отображается любой экран,
   **Then** используется тёмная тема; интерактивные элементы имеют тач-таргеты
   не меньше минимально допустимого размера (44×44 CSS px).
4. **Given** врач проскроллил протокол вниз, **When** он хочет перейти в другую
   секцию, **Then** навигация по секциям доступна без прокрутки к началу
   документа.

### User Story 3 - Найти текст внутри открытого протокола (Priority: P3)

Врач открыл протокол и ищет конкретное слово или значение (например, название
препарата) текстовым поиском по содержимому открытого протокола; совпадения
подсвечиваются, к ним можно перейти.

**Why this priority**: ускоряет работу, но нужный фрагмент можно найти и через
навигацию по секциям (P2), поэтому поиск — усилитель, а не основа сценария.

**Independent Test**: в открытом демо-протоколе ввести строку, встречающуюся в
нескольких блоках, и убедиться, что все совпадения найдены и достижимы.

**Acceptance Scenarios**:

1. **Given** открыт протокол, **When** врач вводит поисковую строку,
   **Then** совпадения в текстовом содержимом протокола подсвечены, и к каждому
   можно перейти.
2. **Given** введена строка в другом регистре, чем в тексте,
   **When** выполняется поиск, **Then** совпадения находятся без учёта регистра.
3. **Given** введена строка без совпадений, **When** выполняется поиск,
   **Then** отображается явное состояние «ничего не найдено» (не пустой экран).

### User Story 4 - Дисклеймер при первом запуске (Priority: P2)

При первом запуске приложение показывает дисклеймер: материал носит справочный
характер, не заменяет официальный протокол и клиническое суждение врача. Доступ
к содержимому — только после явного подтверждения; при последующих запусках
дисклеймер не показывается.

**Why this priority**: обязательное условие безопасного использования
справочника (принцип «справочник, а не советчик»); реализуется независимо от
остальных историй.

**Independent Test**: очистить локальные данные, запустить приложение —
дисклеймер показан; подтвердить, перезапустить — не показан.

**Acceptance Scenarios**:

1. **Given** приложение запускается впервые на устройстве, **When** оно
   открывается, **Then** до любого контента показывается дисклеймер с текстом о
   справочном характере материала, требующий явного подтверждения.
2. **Given** дисклеймер подтверждён, **When** приложение запускается повторно,
   **Then** дисклеймер не показывается, врач сразу попадает на главный экран.

---

### Edge Cases

- Реестр протоколов пуст или не читается: главный экран показывает явное
  состояние «протоколы недоступны», а не пустоту или белый экран.
- Протокол указан в реестре, но его файл отсутствует или содержит невалидный
  JSON: пользователь видит понятное сообщение об ошибке на экране протокола;
  приложение не падает.
- Протокол не проходит валидацию схемы в dev-режиме: разработчик видит ошибку с
  указанием файла и места нарушения схемы; в продакшн-режиме отображается всё,
  что рендерится, а проблемные блоки — через видимый фоллбэк.
- Очень длинный протокол (десятки секций): навигация по секциям остаётся
  доступной и не перекрывает содержимое.
- Строка поиска содержит специальные символы: поиск трактует их буквально и не
  ломается.
- Отметки чек-листа: сбрасываются при закрытии протокола и перезапуске;
  никакой итог по отметкам не вычисляется и не интерпретируется.
- Локальные данные устройства очищены: дисклеймер показывается снова при
  следующем запуске.
- Блок `criteria-list` с неизвестным значением `kind`: рендерится как обычный
  список критериев с видимой пометкой о неопознанной категории, без цветовой
  маркировки «по догадке».

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Главный экран MUST отображать список протоколов из реестра
  (`src/data/protocols/index.json`); для каждого протокола видны название,
  версия и дата источника.
- **FR-002**: Экран протокола MUST отображать метаданные прослеживаемости:
  `version`, `sourceDocument`, `sourceDate`, `lastReviewed`.
- **FR-003**: Экран протокола MUST предоставлять якорную навигацию по секциям
  (оглавление или закреплённые табы), доступную из любой точки прокрутки;
  переход к секции — один тап из видимой навигации.
- **FR-004**: Секции типов «показания», «абсолютные противопоказания»,
  «относительные противопоказания» MUST иметь мгновенно различимую цветовую
  маркировку, заданную единой цветовой темой приложения (не «зашитую» в
  отдельные компоненты).
- **FR-005**: Рендерер MUST отображать блоки типов v1: `text`, `warning`,
  `timer-note`, `criteria-list` (с `kind`: `inclusion` | `exclusion-absolute` |
  `exclusion-relative`), `dosage-table`, `checklist` — исключительно из данных
  протокола, без медицинского текста в коде компонентов.
- **FR-006**: Блок неизвестного типа MUST рендериться видимым фоллбэком
  «неподдерживаемый тип блока» с указанием фактического типа; молчаливый
  пропуск блока запрещён.
- **FR-007**: Блок `checklist` MUST быть интерактивным (отметка пунктов);
  состояние отметок MUST NOT сохраняться между открытиями протокола; никакие
  итоги или интерпретации по отметкам MUST NOT вычисляться.
- **FR-008**: Приложение MUST предоставлять клиентский текстовый поиск по
  содержимому открытого протокола: без учёта регистра, с подсветкой совпадений
  и переходом к ним; поиск за пределами открытого протокола не выполняется.
- **FR-009**: При первом запуске приложение MUST показывать дисклеймер
  («справочный материал, не заменяет официальный протокол и клиническое
  суждение врача») и открывать контент только после явного подтверждения; факт
  подтверждения хранится локально на устройстве; повторно дисклеймер не
  показывается.
- **FR-010**: В комплекте MUST быть демо-протокол, задействующий все типы
  блоков v1 и все три `kind` критериев; весь его контент — явные заглушки
  («Критерий 1», «Пример дозировки»), не имитирующие реальные клинические
  данные.
- **FR-011**: В dev-режиме данные протокола MUST проходить runtime-валидацию
  против схемы; при нарушении разработчик получает понятную ошибку с указанием
  файла и невалидного места.
- **FR-012**: Все экраны MUST использовать тёмную тему по умолчанию;
  интерактивные элементы MUST иметь тач-таргеты не меньше 44×44 CSS px.
- **FR-013**: Функциональность фичи MUST работать без сети: все данные
  поставляются вместе с приложением, сетевые запросы не выполняются.
- **FR-014**: Ошибочные состояния (пустой реестр, отсутствующий файл протокола,
  невалидный JSON) MUST отображаться понятными сообщениями; падение приложения
  или пустой экран недопустимы.
- **FR-015**: После первого открытия с сетью приложение MUST полностью работать
  офлайн при последующих запусках. При выходе новой версии пользователь MUST
  получить явное уведомление с кнопкой применения обновления; молчаливая
  подмена версии и молчаливое залипание на старой версии недопустимы.

### Key Entities

- **Реестр протоколов**: перечень доступных протоколов; для каждой записи —
  идентификатор, название, версия, дата источника и ссылка на файл протокола.
- **Протокол**: единица контента; метаданные прослеживаемости (`version`,
  `sourceDocument`, `sourceDate`, `lastReviewed`) и упорядоченный список секций.
- **Секция**: часть протокола; заголовок, тип (обычная / показания / абсолютные
  противопоказания / относительные противопоказания) и упорядоченный список
  блоков; служит целью якорной навигации.
- **Блок**: атомарная единица отображения одного из типов v1 со своим
  содержимым; тип определяет способ рендеринга.
- **Состояние чек-листа**: эфемерные отметки пунктов в рамках открытого
  протокола; не сохраняется.
- **Подтверждение дисклеймера**: локальный флаг устройства «дисклеймер принят».

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Врач, знакомый с приложением, добирается от запуска приложения до
  нужной секции протокола не более чем за 3 тапа и не более чем за 10 секунд.
- **SC-002**: В проверке на демо-протоколе пользователь верно определяет тип
  секции (показания / абсолютные / относительные противопоказания) по её
  маркировке, не читая заголовок, в 100% случаев.
- **SC-003**: 100% типов блоков v1 демо-протокола отображаются корректно; блок
  неизвестного типа даёт видимую пометку в 100% случаев.
- **SC-004**: В авиарежиме (полное отсутствие сети) все сценарии P1–P3
  выполняются без деградации; количество сетевых запросов — ноль.
- **SC-005**: Поиск по открытому протоколу показывает совпадения менее чем за
  1 секунду после ввода запроса.
- **SC-006**: Дисклеймер показывается при первом запуске в 100% случаев и ни
  разу — после подтверждения (пока локальные данные не очищены).
- **SC-007**: При намеренной порче демо-протокола разработчик локализует ошибку
  по сообщению валидации (файл + место) без чтения кода рендерера.

## Assumptions

- Язык интерфейса и контента — русский.
- Приложение делается под одного конкретного пользователя (жена владельца
  проекта, врач-невролог, не дальтоник): маркировка секций только цветом —
  осознанное решение (YAGNI); при появлении пользователя с иными потребностями
  добавляется альтернативная цветовая тема, а не переделка маркировки.
- Приложение — модульный каркас: добавление и удаление протокола — это только
  JSON-файл плюс запись в реестре; новые типы блоков добавляются как новые
  модули рендерера, не затрагивая существующие; будущие фичи (например,
  ТЛТ-калькулятор) подключаются как отдельные модули поверх каркаса.
- Тёмная тема — единственная обязательная в этой фиче; светлая тема и
  переключатель тем не требуются.
- «Первый запуск» определяется по локальным данным устройства/браузера; после
  их очистки дисклеймер показывается снова — это приемлемо.
- Реестр и файлы протоколов поставляются в составе приложения; добавление
  протокола — новый JSON-файл плюс запись в реестре, без изменения кода
  (конституция, принцип III).
- Полный офлайн входит в эту фичу: после первого открытия с сетью приложение
  работает автономно при последующих запусках (конституция, принцип IV);
  сетевых запросов к внешним сервисам нет вовсе.
- Поиск — точное вхождение подстроки без учёта регистра; морфология, нечёткий
  поиск и поиск по всем протоколам сразу не требуются.
- Блок `timer-note` — статическое визуально акцентированное примечание о
  временном окне с текстом из исходного документа; никакого запускаемого
  таймера или отсчёта времени в нём нет.
- Выбор конкретного паттерна якорной навигации (оглавление против закреплённых
  табов) — решение этапа дизайна/планирования; спецификация требует лишь
  «один тап из видимой навигации в любой точке прокрутки».
- Демо-протокол используется и как фикстура для разработки, и как проверочный
  пример рендерера; в продакшн-реестре он может быть скрыт (решение этапа
  планирования реального контента).
- Вне скоупа: реальный клинический контент (включая ТЛТ — отдельная фича), iOS,
  любые сетевые взаимодействия, аккаунты, аналитика, синхронизация,
  пуш-уведомления (конституция, принцип VI).
