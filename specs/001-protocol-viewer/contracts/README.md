# Контракты данных

Два JSON Schema (draft 2020-12) — контракт между контентом (JSON-файлы,
редактируются вручную) и кодом (генерический рендерер):

- `protocol.schema.json` — документ протокола (`src/data/protocols/<id>.json`)
- `index.schema.json` — реестр (`src/data/protocols/index.json`)

## Где живёт каноник

На этапе реализации схемы копируются в `src/data/schema/` — с этого момента
каноник там (используется dev-валидацией, CI-скриптом и `$schema`-ссылками из
JSON-файлов контента). Файлы здесь — зафиксированный дизайн-артефакт Phase 1.

## Правила совместимости

1. **Неизвестный тип блока валиден** (обязателен только `type: string`).
   Это опора FR-006: старая версия приложения на новых данных показывает
   видимый фоллбэк, а не ошибку. Строгая валидация применяется только к
   известным типам v1.
2. **Новый тип блока** = новая ветка `if/then` в `allOf` + `$defs`-описание +
   вариант TS-union + компонент + строка реестра компонентов. Существующие
   данные не затрагиваются.
3. **`additionalProperties: false`** во всех известных структурах: опечатка в
   имени поля — ошибка валидации, а не молча проигнорированное поле
   (критично для медицинского контента).
4. Поля прослеживаемости (`version`, `sourceDocument`, `sourceDate`,
   `lastReviewed`) — обязательные: протокол без них не проходит валидацию
   (конституция, принцип V).
5. Ячейки `dosage-table` — только строки: контракт не даёт почвы для
   вычислений над дозировками (конституция, принцип I).
