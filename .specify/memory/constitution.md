# Конституция проекта Med Helper

Офлайн-справочник клинических протоколов для практикующего врача.
PWA, публикуемое на GitHub Pages; упаковка в Android APK через Capacitor —
возможное будущее направление, которое архитектура не должна блокировать.

## Core Principles

### I. Справочник, а не советчик

Приложение отображает клинические протоколы в структурированном виде — и не делает
ничего сверх этого.

- ЗАПРЕЩЕНА любая логика, выдающая клиническое решение или рекомендацию:
  «показано / не показано», «назначить / не назначать», автоматический выбор тактики.
- Суммирующие калькуляторы шкал допустимы: они выполняют исключительно арифметическое
  суммирование баллов и возвращают число.
- Программная интерпретация результата (сопоставление суммы с порогами, категориями
  риска, выводами) ЗАПРЕЩЕНА. Интерпретационная таблица может присутствовать только
  как статический текст протокола, отображаемый наравне с остальным контентом.

Обоснование: клиническое решение и ответственность за него остаются за врачом;
приложение — инструмент быстрого доступа к протоколу, а не система поддержки
принятия решений.

### II. Запрет на генерацию медицинского контента (NON-NEGOTIABLE)

- AI-агент НИКОГДА не пишет клинические критерии, дозировки, временные окна,
  противопоказания или иной медицинский контент из собственных знаний.
- Единственный допустимый источник контента — документ, предоставленный владельцем
  проекта. Формулировки переносятся с сохранением исходного текста.
- Неоднозначные, нечитаемые или противоречивые места агент помечает TODO-маркером
  для ручной проверки врачом. Агент не «догадывается» и не дополняет пробелы.

Обоснование: ошибка в дозировке или критерии, привнесённая генерацией, опасна для
пациента и не обнаружима без построчной сверки; запрет делает сверку возможной.

### III. Данные отдельно от кода

- Весь контент протоколов хранится в JSON-файлах в `src/data/protocols/`.
- В компонентах Vue медицинского текста нет: компоненты рендерят любой валидный
  протокол по общей схеме данных.
- Добавление нового протокола НЕ требует изменения кода — только новый JSON-файл
  (и его регистрация в индексе данных, если таковой есть).

Обоснование: врач ревьюит контент как данные, не читая код; код и контент
развиваются и проверяются независимо.

### IV. Полный офлайн

- После первого открытия приложение работает без сети: service worker прекэширует
  все ассеты и все данные протоколов.
- ЗАПРЕЩЕНЫ рантайм-запросы к внешним API, CDN, шрифтам и любым сторонним ресурсам.
- Обновление кэша выполняется только с явным уведомлением пользователя; молчаливая
  подмена версии контента запрещена.

Обоснование: приложение используется в клинических условиях, где сети может не быть;
врач обязан знать, что версия протоколов изменилась.

### V. Прослеживаемость контента

- Каждый протокол обязан нести поля `version`, `sourceDocument`, `sourceDate`,
  `lastReviewed`. Протокол без любого из этих полей не проходит валидацию данных.
- Эти поля всегда видимы в UI при просмотре протокола.

Обоснование: врач должен в любой момент видеть, из какого документа и какой давности
взят протокол, и когда он последний раз проверялся.

### VI. Приватность по умолчанию

- Никаких аккаунтов, аналитики, телеметрии, синхронизации и пуш-уведомлений.
- Приложение не отправляет никаких данных за пределы устройства.

### VII. Работоспособность web-сборки

- `npm run dev` и `npm run build` остаются работоспособными всегда: web-сборка —
  основной инструмент итерации и запасной вариант доставки PWA.
- Никакая интеграция (включая будущий Capacitor) не вправе ломать или усложнять
  web-путь; нативные обвязки подключаются только поверх работающей web-сборки.

## Технологические ограничения

- Платформа: одностраничное приложение на Vue, собираемое в полностью статические
  артефакты; серверной части нет.
- Хостинг: GitHub Pages; сборка обязана корректно работать с настраиваемым базовым
  путём (base URL) статического хостинга.
- Будущая упаковка в Android APK через Capacitor не должна блокироваться
  архитектурой (относительные пути, отсутствие серверных зависимостей, web-first
  структура), но зависимости Capacitor НЕ добавляются, пока упаковка не начата.

## Рабочий процесс и контроль качества

- Каждый spec и plan проходит Constitution Check: функциональность, нарушающая
  принципы I–VII, отклоняется или переформулируется до начала реализации.
- Валидация схемы данных протоколов (включая обязательные поля из принципа V) —
  обязательная часть сборки или тестов.
- Перенесённый контент с TODO-пометками не считается готовым: протокол получает
  актуальный `lastReviewed` только после ручной проверки врачом.
- Прохождение `npm run build` — обязательное условие принятия любых изменений.

## Governance

- Конституция превалирует над всеми прочими практиками и артефактами Spec Kit
  (spec, plan, tasks); при конфликте исправляется артефакт, а не конституция.
- Принципы I, II, IV и VI исключений не допускают. Отступления от остальных
  положений возможны только с явным обоснованием в Complexity Tracking плана.
- Поправки вносятся правкой этого файла с обоснованием и повышением версии по
  semver: MAJOR — удаление или несовместимое переопределение принципа; MINOR —
  новый принцип или существенное расширение; PATCH — уточнения формулировок.
- Проверка соответствия выполняется при каждом `/speckit-plan` (секция
  Constitution Check) и при ревью изменений.
